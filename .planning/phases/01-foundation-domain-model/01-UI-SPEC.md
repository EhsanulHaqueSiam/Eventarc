## Design System

| Property | Value |
|----------|-------|

### shadcn Initialization Instructions

No `components.json` detected. The project is React + Vite. shadcn must be initialized before Phase 1 frontend work begins.

**Steps:**
1. Scaffold frontend: `pnpm create vite@latest frontend --template react-ts`
2. Install TailwindCSS: `cd frontend && pnpm add -D tailwindcss @tailwindcss/vite`
3. Go to [ui.shadcn.com/create](https://ui.shadcn.com/create) and configure a preset. Recommended settings: base style (nova), dark mode support enabled, base-ui primitives.
4. Copy the preset code and run: `pnpm dlx shadcn@latest init --preset <code> --template vite`
5. Verify: `pnpm dlx shadcn@latest info` should show the project config.

## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline element padding, badge internal padding |
| sm | 8px | Compact element spacing, table cell padding, button icon gap |
| md | 16px | Default element spacing, card internal padding, form field gaps |
| lg | 24px | Section padding inside cards, sidebar section gaps |
| xl | 32px | Layout column gaps, page section vertical spacing |
| 2xl | 48px | Major content area breaks, dashboard grid gaps |
| 3xl | 64px | Page-level top/bottom padding, hero-area spacing |

Exceptions:
- **44px minimum touch targets** on all interactive elements in vendor/mobile views (buttons, dropdown triggers, tab triggers). This is a hard requirement for cross-device compatibility (D-28: iPhone, iPad, Android).
- **40px sidebar item height** for comfortable admin navigation.
- **36px** minimum height for all form inputs on desktop; **44px** on mobile.

## Typography

| Role | Size | Weight | Line Height | Usage |
|------|------|--------|-------------|-------|
| Body | 14px | 400 (regular) | 1.5 | Default text, descriptions, table cells, form labels |
| Label | 12px | 500 (medium) | 1.4 | Metadata, captions, badge text, timestamps, breadcrumb items |
| Heading | 20px | 600 (semibold) | 1.3 | Section headings, card titles, dialog titles, page subtitles |
| Display | 28px | 700 (bold) | 1.2 | Page titles, dashboard metric numbers, event names |

**Font stack:**
```css
--font-sans: "DM Sans", ui-sans-serif, system-ui, sans-serif;
--font-mono: "JetBrains Mono", ui-monospace, monospace;
```

**Rules:**
- Body and Label use weight 400-500 only. Never bold body text for emphasis; use `font-medium` (500) sparingly.
- Headings use weight 600. Display uses weight 700.
- Maximum 2 font weights on any single screen (400 + 600 for most views; 400 + 700 for dashboard metric displays).
- Line-height values are non-negotiable. `1.5` for readable body text, `1.2` for display numbers.
- Monospace font reserved for: sizing guide specs, Redis key examples, API endpoint paths, event IDs.

## Color

The color system uses shadcn semantic tokens. Exact hex values depend on the user's chosen preset, but the contract specifies the role distribution and semantic usage.

| Role | Semantic Token | Usage |
|------|---------------|-------|
| Dominant (60%) | `--background` / `--card` | Page backgrounds, main content area, card surfaces |
| Secondary (30%) | `--muted` / `--sidebar` / `--accent` | Sidebar background, navigation, section dividers, table header rows, hover states |
| Accent (10%) | `--primary` | See "Accent reserved for" below |
| Destructive | `--destructive` | Delete event, archive event, remove stall, remove category |
| Success | Custom `--success: oklch(0.72 0.19 142)` | Event "live" status badge, healthy system indicators |
| Warning | Custom `--warning: oklch(0.82 0.17 85)` | Event "draft" status badge, "active" awaiting go-live |

**Accent reserved for (exhaustive list):**
- Primary CTA buttons ("Create Event", "Save Configuration", "Add Category", "Add Stall")
- Active sidebar navigation item indicator
- Event lifecycle state machine active/current state node
- Focus rings on form inputs
- Selected tab underline/indicator
- Toggle/switch active state

**Event lifecycle status badge colors (semantic mapping):**

| Status | Badge Variant | Rationale |
|--------|--------------|-----------|
| draft | `secondary` + warning dot | Not yet ready, needs attention |
| active | `outline` + primary border | Configured, awaiting go-live |
| live | Custom success variant (green) | Actively scanning, most important state |
| completed | `secondary` | Finished, read-only |
| archived | `secondary` + muted text | Historical, low visibility |

**Custom CSS variables to add to the theme file:**
```css
@theme inline {
  --color-success: oklch(0.72 0.19 142);
  --color-success-foreground: oklch(0.98 0.02 142);
  --color-warning: oklch(0.82 0.17 85);
  --color-warning-foreground: oklch(0.25 0.05 85);
}
```

## Interaction Patterns

### Event Creation Flow
1. User clicks "Create Event" (primary CTA on events list page)
2. Dialog opens with form: Name (required), Date (required), Venue (optional), Description (optional)
3. Configuration section in same dialog with ToggleGroups for QR strategy, food QR mode, food QR timing (defaults pre-selected: unified, guestLinked, preSent)
4. "Create Event" button in dialog footer (disabled until required fields valid)
5. On success: toast "Event created", navigate to event detail page
6. Event starts in "draft" status

### Event Lifecycle Transitions
1. State machine visualization shows current state prominently
2. Available transition button(s) below the visualization
3. "Go Live" and "Delete" transitions require AlertDialog confirmation
4. After transition: toast notification, badge updates inline, state machine re-renders

### Vendor Hierarchy CRUD
1. Vendor categories displayed as cards grouped under "Entry Vendors" and "Food Vendors" sections
2. Click "Manage Stalls" on a category card opens a Sheet (side panel) with stall table
3. "Add Stall" opens an inline form row at bottom of table (not a dialog -- reduce friction)
4. "Remove Stall" triggers AlertDialog confirmation
5. All vendor CRUD on event detail page, never separate pages

### Configuration Editing
1. ToggleGroups for each config option (QR strategy, food mode, food timing)
2. Sticky footer bar appears when any config differs from saved state
3. "Discard" resets to saved values
4. "Save Configuration" persists and shows success toast
5. Configuration locked (read-only with explanation) once event is "live"

### Navigation
1. Sidebar is persistent on desktop, collapsible on tablet, Sheet on mobile
2. Breadcrumb always visible in main content area header
3. Event detail uses horizontal Tabs for sub-views (Overview, Configuration, Categories, Vendors)
4. Tab state preserved in URL via TanStack Router search params

## Accessibility Contract

- All interactive elements must have visible focus rings (2px, primary color, 2px offset)
- Color is never the only indicator of state -- always paired with text label or icon
- Event status badges include both color AND text label
- Minimum contrast ratio: 4.5:1 for body text, 3:1 for large text (WCAG AA)
- All dialogs and sheets must have a visible title (use `sr-only` class only if alternative visible heading exists)
- Tab navigation works across all interactive elements
- Form validation errors announced via `aria-invalid` + `aria-describedby`
- Sidebar navigation items have `aria-current="page"` for active route

## Design Principles (from frontend-design skill)

**Aesthetic direction:** Professional SaaS with refined minimalism. Not brutalist, not maximalist -- precision-engineered clarity. The dashboard should feel like Linear or Vercel's admin panel: clean surfaces, deliberate spacing, subtle depth through shadow and border, information density without clutter.

**Differentiation:** The event lifecycle state machine visualization is the memorable element. A horizontal pipeline of connected nodes that shows at a glance where an event stands -- not a boring dropdown, but a visual status tracker that makes the admin feel in control.

**What to avoid:**
- Generic template aesthetics (cookie-cutter cards with no hierarchy)
- Overuse of color (maintain the 60/30/10 discipline)
- Deep nesting of UI elements (keep information architecture flat: max 2 clicks to any feature)
- Tooltips as a crutch for bad labeling (labels should be self-explanatory)
