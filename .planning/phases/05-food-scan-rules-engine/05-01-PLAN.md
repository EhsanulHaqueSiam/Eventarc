<objective>
Implement the food scan processing service with atomic Redis Lua script for consumption check-and-increment. A vendor device POSTs a QR payload to `POST /api/v1/scan/food`, the handler decodes the QR (reusing `qr.DecodePayload`), validates the HMAC signature and QR type (food or unified), determines guest-linked vs anonymous mode from event config, resolves the guest category, and atomically checks consumption against configured limits via a Redis Lua script. Returns structured JSON responses with consumption info for valid scans and consumption history for rejections. Supports both guest-linked and anonymous modes using the same Lua script with different Redis key prefixes.
</objective>

Create `backend/internal/scan/food_lua.go` containing the Redis Lua script for atomic food consumption check-and-increment:

```go
package scan

// foodScanLua is the Redis Lua script for atomic food scan processing.
// It performs: (1) read limit from rules cache, (2) check current consumption,
// (3) increment if allowed, (4) update dashboard counters, (5) log consumption.
//
// KEYS[1] = food:{eventId}:{guestOrTokenId}     (HASH — per-guest/token consumption counts)
// KEYS[2] = foodrules:{eventId}                   (HASH — food rules cache)
// KEYS[3] = counters:{eventId}                    (HASH — dashboard counters)
// KEYS[4] = foodlog:{eventId}:{guestOrTokenId}   (LIST — consumption history log)
//
// ARGV[1] = guestCategoryId (for rule lookup: "{guestCategoryId}:{foodCategoryId}")
// ARGV[2] = foodCategoryId (food category being scanned)
// ARGV[3] = stallId
// ARGV[4] = timestamp (ISO 8601 string)
// ARGV[5] = deviceId
// ARGV[6] = stallName (human-readable, for log entry)
//
// Returns table:
//   [1] = "OK" | "LIMIT_REACHED" | "NO_RULE"
//   [2] = current count (after increment if OK, or current count if rejected)
//   [3] = limit value (-1 for unlimited, 0+ for specific limit)
const foodScanLua = `
local ruleKey = ARGV[1] .. ':' .. ARGV[2]
local limit = redis.call('HGET', KEYS[2], ruleKey)

if limit == false then
  return {'NO_RULE', '0', '0'}
end

limit = tonumber(limit)

if limit == -1 then
  local newCount = redis.call('HINCRBY', KEYS[1], ARGV[2], 1)
  redis.call('HINCRBY', KEYS[3], 'food:' .. ARGV[2] .. ':served', 1)
  redis.call('HINCRBY', KEYS[3], 'food:' .. ARGV[3] .. ':served', 1)
  local logEntry = ARGV[4] .. '|' .. ARGV[3] .. '|' .. ARGV[6]
  redis.call('LPUSH', KEYS[4], logEntry)
  redis.call('LTRIM', KEYS[4], 0, 49)
  return {'OK', tostring(newCount), '-1'}
end

local current = tonumber(redis.call('HGET', KEYS[1], ARGV[2]) or '0')

if current >= limit then
  return {'LIMIT_REACHED', tostring(current), tostring(limit)}
end

local newCount = redis.call('HINCRBY', KEYS[1], ARGV[2], 1)
redis.call('HINCRBY', KEYS[3], 'food:' .. ARGV[2] .. ':served', 1)
redis.call('HINCRBY', KEYS[3], 'food:' .. ARGV[3] .. ':served', 1)

local logEntry = ARGV[4] .. '|' .. ARGV[3] .. '|' .. ARGV[6]
redis.call('LPUSH', KEYS[4], logEntry)
redis.call('LTRIM', KEYS[4], 0, 49)

return {'OK', tostring(newCount), tostring(limit)}
`
```

The script converts all return values to strings because Redis Lua returns integers/strings but go-redis `StringSlice()` expects all strings. The `or '0'` fallback handles the case where `HGET` returns `false` (nil) when no consumption exists yet. The `limit == false` check (not `== nil`) is correct Lua idiom for Redis nil returns.
</action>

<threat_model>
- **T-05-01 (HIGH):** Race condition in consumption check-and-increment — Mitigated by Redis Lua script executing atomically. HGET (read count) + comparison + HINCRBY (increment) run as single atomic operation. No interleaving possible even with 10K concurrent requests for the same guest/category.
- **T-05-02 (HIGH):** Over-serving due to stale count — Mitigated by reading count inside the Lua script (not before it). Every Lua execution sees the most recent state because Redis is single-threaded.
- **T-05-03 (MEDIUM):** Fail-open on missing rules — Mitigated by returning `NO_RULE` which the Go service maps to a rejection. Fail-closed by design: if rules haven't been synced, no food is served.
</threat_model>

```go
    DeviceID       string `json:"device_id"`

type FoodScanResult struct {
    Status       string            `json:"status"`                  // "valid", "limit_reached", "no_rule"
    FoodCategory *FoodCategoryInfo `json:"food_category,omitempty"`
    Consumption  *ConsumptionInfo  `json:"consumption,omitempty"`
    Guest        *GuestInfo        `json:"guest,omitempty"`         // reuse from types.go
    Scan         *ScanInfo         `json:"scan,omitempty"`          // reuse from types.go
    Message      string            `json:"message,omitempty"`
    History      []HistoryEntry    `json:"history,omitempty"`       // populated on rejection
}

// FoodCategoryInfo identifies the food category being scanned.
type FoodCategoryInfo struct {
    ID   string `json:"id"`
    Name string `json:"name"`
}

// ConsumptionInfo shows current consumption state against the limit.
type ConsumptionInfo struct {
    Current   int `json:"current"`    // how many servings consumed so far
    Limit     int `json:"limit"`      // max allowed (-1 for unlimited)
    Remaining int `json:"remaining"`  // how many left (-1 for unlimited)
}

// HistoryEntry is a single consumption record shown in rejection responses.
type HistoryEntry struct {
    StallName  string `json:"stall_name"`
    StallID    string `json:"stall_id"`
    ConsumedAt string `json:"consumed_at"`
}
```

```go
package scan

import (
    "context"
    "fmt"
    "strings"
    "strconv"
    "time"

    "github.com/ehsanul-haque-siam/eventarc/internal/model"
    "github.com/ehsanul-haque-siam/eventarc/internal/qr"
    "github.com/redis/go-redis/v9"
)

var foodScript = redis.NewScript(foodScanLua)

// ProcessFoodScan is the main food scan pipeline:
// 1. Decode QR payload (HMAC verification) via qr.DecodePayload
// 2. Validate QR type is QRTypeFood (0x02) or QRTypeUnified (0x03)
// 3. Determine food mode from event config (Redis: event:{eventId} → foodQrMode field)
// 4. Resolve identity and guest category:
//    - Guest-linked: Redis HGET guest:{eventId}:{guestId} → "category" field
//    - Anonymous: Redis HGET anontoken:{eventId}:{guestId} → "category" field
// 5. Build Redis keys based on mode:
//    - Guest-linked: food:{eventId}:{guestId}
//    - Anonymous: food:{eventId}:anon:{guestId}
// 6. Execute foodScanLua atomically
// 7. Parse Lua result and build FoodScanResult
// 8. On rejection: read consumption log from Redis for history entries
func (s *Service) ProcessFoodScan(ctx context.Context, req FoodScanRequest) (FoodScanResult, error)
```

The method must implement these steps:

**Step 1 — QR Decode:**
```go
payload, err := qr.DecodePayload(req.QRPayload, s.hmacSecret)
```
Return QR-specific errors directly (handler maps them to HTTP codes).

**Step 2 — QR Type Validation:**
```go
if payload.QRType != qr.QRTypeFood && payload.QRType != qr.QRTypeUnified {
    return FoodScanResult{}, fmt.Errorf("%w: expected food or unified QR, got %s", qr.ErrInvalidQRType, qr.QRTypeName(payload.QRType))
}
```

**Step 3 — Determine Food Mode:**
```go
foodQrMode, err := s.redis.HGet(ctx, fmt.Sprintf("event:%s", payload.EventID), "foodQrMode").Result()
```
If Redis returns nil, return error (event not synced). `foodQrMode` is either `"guestLinked"` or `"anonymous"`.

**Step 4 — Resolve Identity and Guest Category:**

For guest-linked mode:
```go
guestKey := fmt.Sprintf("guest:%s:%s", payload.EventID, payload.GuestID)
guestData, err := s.redis.HGetAll(ctx, guestKey).Result()
// guestData["category"] is the guestCategoryId
// guestData["name"] and guestData["categoryLabel"] for response
```

For anonymous mode:
```go
tokenKey := fmt.Sprintf("anontoken:%s:%s", payload.EventID, payload.GuestID)
tokenData, err := s.redis.HGetAll(ctx, tokenKey).Result()
// tokenData["category"] is the guestCategoryId
```

If lookup returns empty map, return `model.ErrNotFound`.

**Step 5 — Build Redis Keys:**

Guest-linked consumption key: `food:{eventId}:{guestId}`
Anonymous consumption key: `food:{eventId}:anon:{guestId}`
Rules key: `foodrules:{eventId}`
Counters key: `counters:{eventId}`
Log key (guest-linked): `foodlog:{eventId}:{guestId}`
Log key (anonymous): `foodlog:{eventId}:anon:{guestId}`

**Step 6 — Execute Lua Script:**
```go
now := time.Now().UTC().Format(time.RFC3339)
result, err := foodScript.Run(ctx, s.redis,
    []string{consumptionKey, rulesKey, countersKey, logKey},
    guestCategoryID, req.FoodCategoryID, req.StallID, now, req.DeviceID, stallName,
).StringSlice()
```

**Step 7 — Parse Result:**
- `result[0] == "OK"`: Build FoodScanResult with status "valid", parse result[1] as current count, result[2] as limit. Compute remaining.
- `result[0] == "LIMIT_REACHED"`: Build FoodScanResult with status "limit_reached", parse counts. Fetch history from log key.
- `result[0] == "NO_RULE"`: Return error wrapping `model.ErrNotFound` with message "no food rule configured for this combination".

**Step 8 — On Rejection, Read History:**
```go
entries, _ := s.redis.LRange(ctx, logKey, 0, 9).Result()
// Parse pipe-delimited entries: "timestamp|stallId|stallName"
```

Also add a helper method to the Service:
```go
// GetStallName retrieves human-readable stall name from Redis cache.
// Key: stall:{eventId}:{stallId} → "name" field
func (s *Service) GetStallName(ctx context.Context, eventID, stallID string) string
```
Falls back to stallID if not cached.
</action>

```go
package scan

import (
    "encoding/json"
    "errors"
    "log/slog"
    "net/http"

    "github.com/ehsanul-haque-siam/eventarc/internal/model"
    "github.com/ehsanul-haque-siam/eventarc/internal/qr"
)

// HandleFoodScan returns an HTTP handler for POST /api/v1/scan/food.
// It decodes the JSON request body, calls Service.ProcessFoodScan,
// and returns the appropriate HTTP status and JSON response.
//
// Status codes:
//   200 — Valid food scan (consumption allowed) OR limit reached (business response, not error)
//   400 — Malformed request body or invalid QR payload format
//   401 — Invalid HMAC signature (forged QR)
//   404 — Guest/token not found in cache
//   422 — Wrong QR type (entry QR at food stall)
//   500 — Internal server error (Redis/PG connectivity, event not synced)
func HandleFoodScan(svc *Service) http.HandlerFunc
```

The handler must:
1. Decode JSON body into `FoodScanRequest`. If decode fails, return 400 with `{"error":{"code":"BAD_REQUEST","message":"Invalid request body"}}`.
2. Validate required fields: `qr_payload` non-empty, `stall_id` non-empty, `device_id` non-empty, `food_category_id` non-empty. Return 400 with `{"error":{"code":"BAD_REQUEST","message":"Missing required field: {field}"}}` if missing.
3. Call `svc.ProcessFoodScan(r.Context(), req)`.
4. If no error: return 200 with `FoodScanResult` JSON (covers both valid and limit_reached — both are business responses).
5. Map errors to HTTP status codes:
   - `qr.ErrInvalidPayload` or `qr.ErrUnsupportedVersion` → 400 `INVALID_QR`
   - `qr.ErrInvalidSignature` → 401 `INVALID_SIGNATURE`
   - `model.ErrNotFound` → 404 `NOT_FOUND`
   - `qr.ErrInvalidQRType` → 422 `WRONG_QR_TYPE`
   - All other errors → 500 `INTERNAL_ERROR`
6. Log each scan with slog: `slog.Info("food scan", "event_id", eventID, "stall_id", req.StallID, "status", result.Status)` on success. `slog.Warn("food scan error", "error", err)` on failure.

Key design choice: `limit_reached` returns HTTP 200 (not 4xx) because it's a valid business response — the scan was processed correctly, the guest simply hit their limit. The vendor device parses the response body for display regardless of HTTP status. This differs from Phase 4 entry scans where duplicate (409) is a system-level concern.
</action>
