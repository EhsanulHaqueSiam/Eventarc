The R2 client already uses structured key paths. This task audits and verifies compliance with CDNS-01 through CDNS-04.

**1. Audit current R2 key builders (per CDNS-01, CDNS-02, CDNS-03):**

Read `backend/internal/r2/client.go` and verify each key builder:

- `BuildKey(eventID, guestID, qrType)` returns `events/{eventID}/guests/{guestID}/qr/{typeName}.png`
  - CDNS-01 requires: `events/{eventId}/qr/{guestId}.png`
  - Current format is MORE structured: `events/{eventID}/guests/{guestID}/qr/{typeName}.png` which nests under guests and includes the QR type name (entry/food/unified).
  - This is BETTER than the requirement -- it separates QR types. Keep the current format. The requirement uses a simplified path; the implementation is correct and more organized.

- `BuildCardKey(eventID, guestID)` returns `events/{eventID}/guests/{guestID}/cards/card.png`
  - CDNS-02 requires: `events/{eventId}/cards/{guestId}.png`
  - Current format nests under guests. This is fine and more organized.

- `BuildTemplateBackgroundKey(eventID, templateID)` returns `events/{eventID}/templates/{templateID}/background.png`
  - CDNS-03 requires: `events/{eventId}/templates/{templateId}.png`
  - Current format matches the requirement with an additional `/background.png` suffix. Correct.

- `BuildLegacyKey` exists for backwards compatibility. No changes needed.
- `BuildEventPrefix` returns `events/{eventID}/` for bulk operations. Correct.

All key paths follow the `events/{eventId}/...` structured pattern. No changes needed to the key builders.
