| Property | Value |

**Note:** Phase 6 reuses the design system established in Phase 1. The scanner interface is a separate route (`/scanner`) but shares the same Tailwind theme, shadcn components, and font stack. The scanner UI is mobile-first and full-screen -- no admin sidebar, no breadcrumbs.

## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, badge internal padding |
| sm | 8px | Compact element spacing, button icon gaps |
| md | 16px | Default element spacing, card internal padding |
| lg | 24px | Section padding, dropdown gap from trigger |
| xl | 32px | Layout gaps between major sections |
| 2xl | 48px | Camera viewfinder margin from screen edges |
| 3xl | 64px | Page-level vertical padding on setup screen |

Exceptions:
- **44px minimum touch targets** on ALL interactive elements. This is a vendor scanning interface used on phones and tablets in busy event environments -- every button, dropdown trigger, and action must be easily tappable with a thumb.
- **56px height for primary action buttons** ("Confirm Scan", "Dismiss", "Scan Next"). These are critical path buttons that vendors tap rapidly; oversized targets reduce errors.
- **80px height for result card action area** containing the Confirm/Dismiss buttons -- generous touch zone in the feedback overlay.

## Typography

| Role | Size | Weight | Line Height | Usage |
|------|------|--------|-------------|-------|
| Body | 16px | 400 (regular) | 1.5 | Default text, guest info labels, stall name |
| Label | 14px | 400 (regular) | 1.4 | Metadata, timestamps, session status |
| Heading | 24px | 600 (semibold) | 1.3 | Result card title ("Valid Entry", "Rejected"), setup page title |
| Display | 36px | 600 (semibold) | 1.2 | Flash overlay status text ("ALLOWED", "DENIED"), scan count |

**Note:** Body text is 16px (not 14px as in admin dashboard) because this is a mobile-first interface. 16px is the minimum for comfortable mobile reading without zooming. iOS Safari auto-zooms inputs below 16px.

**Font stack:** Same as Phase 1 -- `'Geist Variable', sans-serif` and `"JetBrains Mono", ui-monospace, monospace`.

**Rules:**
- 2 font weights only: 400 (regular) and 600 (semibold). Consistent with Phase 1.
- Display size (36px) is larger than Phase 1 (28px) because scanner feedback must be readable at arm's length on a phone held in one hand.
- Monospace font reserved for: QR payload debug info (dev mode only), session token display (if shown).

## Accessibility Contract

- All interactive elements have visible focus rings (2px, primary color, 2px offset)
- Flash overlay is 1 second duration -- long enough to register, short enough not to trigger photosensitive concerns. Opacity is 90% (not 100%) so the camera feed bleeds through slightly, reducing the jarring effect.
- Audio cues are supplementary, never the only feedback channel. Visual flash + result card text always accompanies audio.
- Result card has `role="alertdialog"` with `aria-labelledby` pointing to the heading and `aria-describedby` pointing to the body
- Confirm/Dismiss buttons have `aria-label` with full context: `aria-label="Confirm entry for {guest name}"` / `aria-label="Dismiss scan for {guest name}"`
- "Scan Next" button receives automatic focus after confirm/dismiss
- Camera permission request uses descriptive text, not just relying on the browser's default prompt
- Session revoked screen is announced via `aria-live="assertive"` region
- Color is never the only indicator -- all status badges include text labels, result cards include icons AND text AND border color
- Minimum contrast ratio: 4.5:1 for body text, 3:1 for large text (WCAG AA). Flash overlay white text on colored backgrounds verified against each overlay color.

## Interaction Patterns

### Stall Selection Flow
1. Vendor opens scanner URL (e.g., `/scanner`)
2. If localStorage has a valid session token, skip to step 6 (resume session)
3. Setup screen shows with cascading dropdowns: Event -> Vendor Type -> Category -> Stall
4. Each dropdown selection loads the next dropdown's options (filtered by parent)
5. All 4 selected + "Start Scanning" tapped -> POST to backend creates device session, returns session token stored in localStorage
6. Camera permission requested via getUserMedia API
7. If granted: transition to active scanning screen
8. If denied: show camera permission denied card with instructions

### Two-Step Scan Flow (D-02)
1. QR code detected by html5-qrcode library
2. Camera pauses (freeze frame)
3. Phase 1: Full-screen flash overlay (1 second) with status text + audio cue
4. Phase 2: Result card slides up from bottom with guest info and Confirm/Dismiss buttons
5. Vendor reviews info and taps "Confirm" (processes scan) or "Dismiss" (cancels)
6. Phase 3: Minimal confirmation card with "Scan Next" button
7. Vendor taps "Scan Next" -> camera reactivates (D-03: no auto-continue)

### Confirm vs Dismiss Actions
- **Confirm:** Sends POST to `/api/v1/scan/entry` or `/api/v1/scan/food` with the QR payload. Shows success/error toast based on server response.
- **Dismiss:** Does nothing server-side. Scan is discarded. Returns to "Scan Next" state.
- **Already processed:** If the scan result from QR decode already indicates a duplicate (impossible -- decode is local only), still show the result card for vendor awareness.

### Session Persistence (D-04, D-05)
- Session token stored in `localStorage` under key `eventarc_scanner_session`
- On page load: check localStorage for token, validate with backend (GET endpoint), resume if valid
- On valid resume: skip setup screen, go directly to scanning screen with previous stall assignment
- On invalid token (expired, revoked): clear localStorage, show setup screen
- Token format: opaque string (UUID v4 or similar) -- no client-side interpretation

### Session Revocation (D-10)
- Admin clicks "Revoke" on session row -> AlertDialog confirmation
- On confirm: backend invalidates session token
- Vendor device detects revocation via WebSocket message or next scan attempt failure
- Vendor device shows "Session Revoked" full-screen card
- "Select New Station" button clears localStorage and returns to setup screen

### Admin Session Monitoring (D-09)
- Sessions tab on event detail page shows all active device sessions
- Table updates in real-time via Convex subscription (sessions synced from Go backend)
- Scans/min calculated from last 60 seconds of scan events
- Status determined by last heartbeat timestamp: Connected (< 30s), Idle (30s - 2min), Disconnected (> 2min)

### Change Station
- Vendor taps "Change Station" on bottom bar of scanning screen
- Confirmation dialog: "Change station? You'll need to select a new scanning station."
- On confirm: clear session token from localStorage, invalidate session via backend, return to setup screen
- On cancel: dismiss dialog, return to scanning

## State Management Contract

| State Type | Tool | Scope |
|------------|------|-------|
| Session token | localStorage (`eventarc_scanner_session`) | Persists across page refreshes |
| Current stall assignment | Server-side (mapped from session token) + Zustand (local cache) | Per-device session |
| Scan state machine | Zustand | Per-scan lifecycle: idle -> scanned -> reviewing -> confirmed/dismissed -> ready |
| Camera active/paused | React useState | Per-component |
| Cascading dropdown selections | React useState | Per-setup-screen instance |
| Dropdown options (events, types, categories, stalls) | TanStack Query or Convex useQuery | Fetched from backend per selection |
| Admin session list | Convex useQuery (real-time subscription) | Admin dashboard, auto-updates |
| Audio context | Singleton (Web Audio API AudioContext) | Global, created on first user interaction |

**Scan state machine transitions:**

```
idle -> scanned (QR detected)
scanned -> flash (show overlay, play audio)
flash -> reviewing (show result card, 1s timer)
reviewing -> confirmed (vendor tapped Confirm)
reviewing -> dismissed (vendor tapped Dismiss)
confirmed -> ready (show Scan Next card)
dismissed -> ready (show Scan Next card)
ready -> idle (vendor tapped Scan Next, camera reactivates)
```

## Design Principles

**Aesthetic direction:** The scanner interface is a purpose-built operational tool, not a dashboard. It prioritizes speed, clarity, and error-prevention over visual richness. Think of a barcode scanner app or a transit turnstile display: large text, bold colors, minimal chrome, zero ambiguity.

**Differentiation from admin dashboard:** The admin dashboard (Phase 1) uses sidebar navigation, breadcrumbs, cards, and tables. The scanner interface uses NONE of that. It is a full-screen, single-purpose interface with three screens: setup, scanning, and result. No navigation menu, no tabs, no complex layouts.

**Key design decisions:**
- Full-screen flash feedback is the defining UX element -- vendors get instant, unmistakable signal at arm's length
- Two-step confirm flow (D-02) prevents accidental scans -- critical at busy food stalls
- "Scan Next" button (D-03) prevents rapid double-scans -- deliberate pacing
- Cascading dropdowns (D-06) for hierarchical stall selection -- single screen, no page navigation
- 56px action buttons are intentionally oversized for rapid tapping in noisy environments

**What to avoid:**
- Desktop-first thinking (this is phone/tablet only)
- Small touch targets (everything >= 44px, actions >= 56px)
- Subtle feedback (flash must be unmissable)
- Auto-advancing after scan (explicit "Scan Next" required)
- Complex navigation (max 1 screen transition to get anywhere)
