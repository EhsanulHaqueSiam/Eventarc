## Interaction Patterns

### Card Editor Flow
1. Admin navigates to event detail, clicks "Card Editor" tab (or sidebar link)
2. Empty canvas state shows "Upload Design Image" button centered in canvas area
3. Admin uploads a PNG/JPG design image via file picker
4. Image renders as Fabric.js canvas background (non-draggable, locked)
5. QR overlay placeholder (sample QR image) auto-placed at center of canvas
6. Admin drags and resizes QR overlay using Fabric.js transform handles
7. Right sidebar shows live X/Y/W/H coordinates updating as QR is moved
8. Admin can type exact coordinates in sidebar inputs to fine-tune position
9. "Preview Composite" shows a thumbnail of the final composite (sample guest QR overlaid on design)
10. "Save Template" saves the design image URL + QR position/size config as a named template

### Template Management
1. "Load Template" opens dialog with grid of saved templates (thumbnail + name + dimensions)
2. Click "Apply" loads the template's design image and QR position into the canvas
3. "Delete" triggers AlertDialog confirmation
4. Templates are event-independent -- saved globally for the admin

### Compositing Pipeline Monitoring
1. Admin clicks "Generate Cards" on the SMS Delivery page
2. Button disables, progress bar appears with real-time count updates
3. Progress updates via polling or SSE (same pattern as Phase 3 QR generation: D-06)
4. Admin can navigate away and return -- progress persists via asynq job state
5. On completion: success toast, progress bar shows 100%, "Generate Cards" button becomes "Regenerate Cards"

### SMS Delivery Flow
1. "Send Invitations" button is disabled until card generation is complete
2. Admin clicks "Send Invitations" -- no confirmation dialog (it is a high-intent action button)
3. Stats row shows real-time delivery counts (queued/sent/delivered/failed)
4. Guest status table updates as delivery events arrive
5. If failures exist: "Retry Failed" button appears, retries only failed messages
6. "Cancel SMS" option available in dropdown menu (not primary button) for stopping in-progress batch

### Navigation
1. Card Editor and SMS Delivery are tabs on the event detail page (extend existing tab navigation)
2. New tabs: "Card Editor" (after existing tabs), "SMS" (after Card Editor)
3. Tab state preserved in URL via TanStack Router search params (consistent with Phase 1 pattern)

## Accessibility Contract

- Canvas area has `role="application"` with `aria-label="Invitation card editor"`
- QR overlay has keyboard support: arrow keys move by 1px, Shift+arrow by 10px
- File upload input is labeled with `aria-label="Upload invitation card design image"`
- Progress bar uses `role="progressbar"` with `aria-valuenow`, `aria-valuemin`, `aria-valuemax`
- SMS status badges include both color AND text label (never color-only)
- All dialogs have visible title (use `DialogTitle` component)
- Tab navigation works across canvas toolbar, sidebar inputs, and action buttons
- Status filter tabs have `aria-controls` pointing to the filtered table
- Retry buttons in table rows have `aria-label="Retry SMS for {guest name}"`
- Minimum contrast ratio: 4.5:1 for body text, 3:1 for large text (WCAG AA)

## State Management Contract

| State Type | Tool | Scope |
|------------|------|-------|
| Server state (templates, SMS status, compositing jobs) | Convex React client (`useQuery`, `useMutation`) | Global, real-time via Convex subscriptions |
| Canvas state (Fabric.js objects, zoom, selection) | Fabric.js internal state | Per-editor instance, not persisted to server |
| QR overlay position/size | React `useState` synced with Fabric.js events | Per-editor, saved to template on "Save" |
| Compositing job progress | TanStack Query polling (or Convex subscription) | Per-event, real-time updates |

## Design Principles

**Aesthetic direction:** Consistent with Phase 1 -- professional SaaS with refined minimalism. The card editor is the memorable element of this phase: a spacious canvas workspace with a clean sidebar for precision controls, reminiscent of Figma's inspector panel but scoped to a single operation (QR positioning).

**Canvas editor feel:** The editor should feel focused and uncluttered. A light muted background behind the canvas creates visual separation. The toolbar is minimal (upload, zoom, fit, reset) -- not a full design tool. The sidebar is for precision, not exploration.

**Progress monitoring:** Compositing and SMS progress should feel reassuring at scale. Large, clear progress bars with numeric counts (not just percentages). Stats cards use bold display typography for the numbers. The goal is to make 60K feel manageable, not overwhelming.

**What to avoid:**
- Overloading the editor toolbar (this is NOT a general-purpose design tool)
- Putting SMS delivery on the same page as the card editor (separate concerns)
- Using modals for progress monitoring (progress should be inline, persistent)
- Hiding the "Retry Failed" action (it should be prominent when failures exist)
