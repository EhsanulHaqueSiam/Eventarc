# Phase 7 — UI Design Contract

> Visual and interaction contract for the Offline Resilience phase. Generated by gsd-ui-researcher, verified by gsd-ui-checker. Extends the Phase 6 scanner interface with offline state indicators, queued scan management, and retroactive rejection notifications.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | shadcn (initialized, base-nova preset) |
| Preset | `base-nova` with Vite template |
| Component library | base-ui primitives (via shadcn) |
| Icon library | lucide-react |
| Font | Geist Variable (body + headings) + JetBrains Mono (data/codes) |

**Note:** Phase 7 reuses the Phase 6 scanner design system entirely. No new fonts, no new base colors, no new shadcn components to install. All additions are visual state overlays and notification patterns layered onto the existing scanner UI. The offline resilience layer is invisible when the device is online.

- Offline banner has `role="status"` with `aria-live="polite"` -- announces "Offline, scans will be queued" to screen readers when network drops
- Connection restored banner has `aria-live="polite"` -- announces "Back online, syncing N scans"
- Pending count badge has `aria-label="N scans pending sync"` (dynamic count)
- Retroactive rejection toast has `role="alertdialog"` with `aria-live="assertive"` -- immediate screen reader announcement
- "Acknowledge" button on rejection toast has `aria-label="Acknowledge rejected scan for {guest name}"`
- Pending queue panel has `role="dialog"` with `aria-labelledby` pointing to the panel title
- Each scan item in pending queue has `aria-label="{guest name}, {scan type}, {status}"` for screen reader traversal
- Focus management: when rejection toast appears, do NOT steal focus from camera/scan flow -- vendor may be mid-scan. Focus moves to toast only when vendor taps it.
- Audio cue for queued scan (330Hz neutral tone) is distinct from all Phase 6 audio cues -- vendor can distinguish queued vs valid vs rejected by sound alone
- Color is never the only indicator: offline banner has text + icon, pending badge has text + icon + count, rejection toast has text + icon + border color + heading

## Interaction Patterns

### Network State Detection
1. Primary detection: monitor WebSocket connection to Go backend (if Phase 6 uses WebSocket for session sync). WebSocket `close` event = offline, `open` event = online.
2. Fallback detection: `navigator.onLine` property + `online`/`offline` window events.
3. Heartbeat: if WebSocket is not used, periodic fetch to a lightweight health endpoint (e.g., GET `/api/v1/health`) every 10 seconds when idle. Failure after 2 consecutive misses = offline state.
4. Network state stored in Zustand scanner store as `networkStatus: 'online' | 'offline' | 'syncing'`.

### Offline Scan Flow (D-01, D-02)
1. Device detects network loss -> offline banner slides down, session indicator changes to warning dot
2. Vendor scans QR -> QR decoded locally (html5-qrcode runs client-side, no network needed)
3. Phase 1: Amber flash overlay with "QUEUED" text + neutral audio cue (330Hz, 150ms)
4. Phase 2: Queued result card with "Queue Scan" / "Dismiss" buttons and guest info
5. Vendor taps "Queue Scan" -> scan payload + idempotency key + timestamp stored in IndexedDB
6. Pending badge count increments in bottom bar
7. Phase 3: Minimal card with "Scan Next" button
8. Vendor taps "Scan Next" -> camera reactivates, ready for next offline scan

### Idempotency Key Generation (D-06)
- Format: `{eventId}-{scanType}-{qrPayloadHash}-{timestamp}-{random4}` (UUID v4 is also acceptable)
- Generated client-side BEFORE writing to IndexedDB
- Same key used when syncing to server -- INSERT ON CONFLICT prevents double-counting
- Key stored alongside scan payload in IndexedDB record

### Reconnection Sync Flow (D-04)
1. Device detects network restored -> offline banner crossfades to connection restored banner: "Back online -- syncing N scans"
2. Network status changes to `syncing`
3. Pending scans retrieved from IndexedDB, sorted by timestamp (oldest first -- D-04)
4. Each scan sent sequentially to server (POST `/api/v1/scan/entry` or `/api/v1/scan/food`) with its idempotency key
5. Server validates each scan against current state:
   - Valid: server processes scan, returns success -> scan marked `synced` in IndexedDB
   - Already processed (idempotency hit): server returns success (idempotent) -> scan marked `synced` in IndexedDB
   - Rejected (limit reached, already checked in by someone else while offline): server returns rejection -> scan marked `rejected` in IndexedDB
6. For each rejected scan: retroactive rejection toast appears (D-05)
7. After all scans processed: connection restored banner slides up and disappears, pending badge disappears, network status returns to `online`
8. Synced and rejected scans remain in IndexedDB for 30 minutes (D-07) for the pending queue panel history, then auto-cleaned

### Retroactive Rejection Notification (D-05)
1. During sync, server returns rejection for a queued scan
2. Rejection toast slides in from top with descending audio cue
3. Toast displays: heading "Scan Rejected Retroactively" + body with guest name and reason
4. Toast does NOT auto-dismiss -- vendor must tap "Acknowledge" (D-05)
5. Multiple rejections stack vertically (each is a separate toast)
6. "Acknowledge" removes the toast. Scan stays marked `rejected` in IndexedDB for pending queue panel visibility.
7. Rejection toasts do NOT interrupt scanning flow -- vendor can continue scanning while toasts are visible

### Pending Queue Panel Interaction
1. Vendor taps pending badge in bottom bar -> panel slides up from bottom
2. Panel shows all scans currently in IndexedDB (pending, synced, rejected)
3. Scans sorted by timestamp (newest first for review)
4. Each scan shows: guest name, scan type, relative time, status badge
5. Close panel: tap X, tap outside, or swipe down
6. No actions available on individual scans -- panel is informational only
7. Panel auto-updates as sync progresses (scans transition from pending to synced/rejected)

### 30-Minute Retention Cleanup (D-07)
1. On every sync completion and every 5 minutes (whichever comes first): scan IndexedDB for records older than 30 minutes
2. Delete records where `timestamp < now - 30min` regardless of status (pending, synced, rejected)
3. If device has been offline > 30 minutes: scans older than 30 min are deleted before sync. This means only the most recent 30 minutes of queued scans are synced -- older scans are lost. This is by design (D-07: 30+ minutes offline indicates a serious problem).
4. Cleanup runs silently -- no UI notification for expired scans

## State Management Contract

| State Type | Tool | Scope |
|------------|------|-------|
| Network status | Zustand (`networkStatus: 'online' \| 'offline' \| 'syncing'`) | Global scanner state |
| Offline scan queue | IndexedDB (via idb or Dexie.js wrapper) | Persistent across page refreshes |
| Pending scan count | Zustand (derived from IndexedDB query) | Reactive UI updates |
| Sync progress | Zustand (`syncProgress: { total: number, completed: number, failed: number }`) | During reconnection sync |
| Retroactive rejections | Zustand (array of rejection details, cleared on acknowledge) | Toast notification queue |
| Pending queue panel open | React useState | Per-component UI state |

**IndexedDB Schema (D-08):**

```
Store: "offline_scans"
Key: idempotency_key (string, primary key)

Fields:
- idempotency_key: string (primary key, client-generated)
- scan_payload: string (raw QR payload for server submission)
- scan_type: "entry" | "food" (determines which endpoint to call)
- stall_id: string (current stall assignment from session)
- event_id: string (current event from session)
- guest_name: string (decoded from QR for display -- may be empty if decode fails offline)
- guest_category: string (decoded from QR for display)
- timestamp: number (Date.now() when scan was queued)
- status: "pending" | "synced" | "rejected" (updated during sync)
- rejection_reason: string | null (populated when status = "rejected")
- synced_at: number | null (timestamp when sync completed)

Indexes:
- by_status: on "status" field (query pending scans for sync)
- by_timestamp: on "timestamp" field (sort for sequential sync, cleanup)
```

**Extended scan state machine (adds offline states to Phase 6):**

```
idle -> scanned (QR detected)
scanned -> flash (show overlay, play audio)

[If online:]
flash -> reviewing (show result card, 1s timer)
reviewing -> confirmed (vendor tapped Confirm)
reviewing -> dismissed (vendor tapped Dismiss)
confirmed -> ready (show Scan Next card)
dismissed -> ready (show Scan Next card)

[If offline:]
flash -> queued_reviewing (show queued result card, 1s amber flash)
queued_reviewing -> queued (vendor tapped Queue Scan, saved to IndexedDB)
queued_reviewing -> dismissed (vendor tapped Dismiss, nothing saved)
queued -> ready (show Scan Next card)
dismissed -> ready (show Scan Next card)

ready -> idle (vendor tapped Scan Next, camera reactivates)
```

## Design Principles

**Aesthetic direction:** The offline resilience layer is an invisible safety net. When the device is online, zero Phase 7 UI elements are visible. When offline, the UI communicates three things clearly: (1) you are offline, (2) scans are being queued, (3) here is how many are pending. The goal is confidence, not anxiety -- the vendor should feel safe continuing to scan.

**Key design decisions:**
- Amber/warning color for all offline states -- distinct from green (valid) and red (rejected). Amber means "noted, will validate later" -- honest, not optimistic (D-01).
- "QUEUED" flash text is unambiguous -- vendor knows this is not confirmed (D-01).
- Rejection toasts are persistent (no auto-dismiss) -- vendor must acknowledge that a previously queued scan was retroactively rejected (D-05).
- Sequential sync preserves temporal ordering -- vendor sees scans resolve in the order they were made (D-04).
- 30-minute retention is aggressive by design -- offline is a brief safety net, not a sustained operating mode (D-07, OFFL-04).

**What to avoid:**
- Optimistic green flash for offline scans (NEVER -- D-01 explicitly prohibits this)
- Auto-dismissing rejection toasts (vendor must acknowledge each one -- D-05)
- Complex sync settings or manual sync triggers (sync is automatic on reconnect)
- Showing too much technical detail to vendor (idempotency keys, IndexedDB state -- dev mode only)
- Blocking the scan flow for sync status (vendor can keep scanning during sync)
