<objective>
Offline scan infrastructure: IndexedDB queue service, network detection hook, and offline scanner UI components.

Purpose: Build the foundation for offline resilience -- when the device loses network, scans are queued locally in IndexedDB with idempotency keys and timestamps. The vendor sees a clear "QUEUED" state (not optimistic approval per D-01), an offline banner, and can continue scanning without interruption. The 30-minute retention window auto-cleans old scans (D-07). This plan delivers the data layer and offline-specific UI; Plan 02 handles sync-on-reconnect and rejection notifications.
Output: IndexedDB service module, network detection hook, Zustand store extensions, three scanner UI components (OfflineBanner, QueuedScanFlash, QueuedScanResultCard), and comprehensive unit tests.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client IndexedDB -> Server scan endpoints | Queued scans are re-validated server-side on sync (OFFL-02). Client cannot bypass server validation. |
| Client-generated idempotency keys -> Server INSERT ON CONFLICT | Client generates keys, server enforces uniqueness. Malicious keys are handled by the server's ON CONFLICT clause. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-01 | Tampering | IndexedDB scan records | accept | IndexedDB is client-side storage. A malicious user could modify queued scans, but server re-validates every scan on sync (OFFL-02). No client-side trust. |
| T-07-02 | Spoofing | Idempotency keys | mitigate | Keys are UUIDs from crypto.randomUUID() (cryptographically secure). Server uses INSERT ON CONFLICT -- duplicate keys are idempotent, not exploitable. |
| T-07-03 | Denial of Service | IndexedDB storage filling | mitigate | 30-minute retention window (D-07) auto-cleans old records. Each scan is ~500 bytes. 30 minutes of scanning at max rate cannot exceed browser quota. |
| T-07-04 | Information Disclosure | Guest names in IndexedDB | accept | Guest names are already visible to the vendor on the scanner screen (Phase 6). IndexedDB storage does not increase exposure beyond what the operator already sees. |
</threat_model>
