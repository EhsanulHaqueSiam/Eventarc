<objective>
Reconnection sync engine, retroactive rejection notifications, and pending queue UI.

Purpose: When the device comes back online, queued scans sync sequentially to the server. Scans rejected during sync (limit reached while offline, already checked in) trigger persistent notification toasts that the vendor must acknowledge (D-05). The vendor can view all pending/synced/rejected scans in a slide-up panel. This completes the offline resilience feature by closing the feedback loop: offline queue (Plan 01) -> sync + notify (Plan 02).

             - `onProgress`: call `setSyncProgress(progress)`
             - `onNetworkError`: set networkStatus back to 'offline'
           - On completion: set networkStatus to 'online', setSyncProgress to null, refresh pendingCount via `getPendingCount()`

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Offline scan queue -> Server sync | Queued scans cross from untrusted client storage to server. Server re-validates every scan (OFFL-02). |
| Session token in sync requests | X-Session-Token header sent with each sync request. Server validates session is active. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-05 | Tampering | Queued scan payload modification | mitigate | QR payload contains HMAC-SHA256 signature (from Phase 3). Server verifies HMAC during sync -- modified payloads are rejected. Client-side tampering of IndexedDB records cannot bypass server HMAC validation. |
| T-07-06 | Repudiation | Sync timing manipulation | accept | Client sends `queued_at` timestamp which is informational. Server uses its own timestamp for the authoritative scan record. Client timestamp manipulation has no security impact. |
| T-07-07 | Denial of Service | Flood sync with fake scans | mitigate | Session token required for sync. Server rate limits per session. HMAC validation rejects forged payloads before any state changes. |
| T-07-08 | Elevation of Privilege | Replay old synced scans | mitigate | Server INSERT ON CONFLICT with idempotency key. Replaying already-synced scans returns success but does not duplicate state. No privilege escalation possible. |
</threat_model>
