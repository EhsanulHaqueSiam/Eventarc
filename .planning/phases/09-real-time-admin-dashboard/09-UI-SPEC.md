## Spacing Scale

Inherited from Phase 1. All values are multiples of 4.

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, status indicator padding |
| sm | 8px | Metric card internal gaps, alert feed item padding |
| md | 16px | Default card padding, metric grid gaps |
| lg | 24px | Section padding, card group spacing |
| xl | 32px | Dashboard section vertical spacing |
| 2xl | 48px | Not used in dashboard (compact layout) |
| 3xl | 64px | Not used in dashboard (compact layout) |

Exceptions:
- **12px** gap between metric value and label inside counter cards (tighter coupling of number-to-label).
- **20px** vertical gap between alert feed items (balance density with readability).

## Typography

Inherited from Phase 1 with dashboard-specific additions:

| Role | Size | Weight | Line Height | Usage |
|------|------|--------|-------------|-------|
| Body | 14px | 400 (regular) | 1.5 | Alert messages, stall names, descriptions |
| Label | 12px | 400 (regular) | 1.4 | Counter labels, timestamps, rate labels |
| Heading | 20px | 600 (semibold) | 1.3 | Section headings ("Attendance", "Food Consumption") |
| Display | 28px | 600 (semibold) | 1.2 | Primary counter values (checked-in count, total scans) |
| Metric | 36px | 600 (semibold) | 1.1 | Hero attendance counter (largest number on dashboard) |

**Font stack:**
```css
--font-sans: "DM Sans", ui-sans-serif, system-ui, sans-serif;
--font-mono: "JetBrains Mono", ui-monospace, monospace;
```

**Dashboard-specific rules:**
- Counter numbers use `--font-mono` for tabular alignment (monospace digits prevent layout shifts as numbers change).
- The hero attendance counter uses the Metric role (36px) -- the single largest element on the dashboard.
- Rate values (e.g., "12/min") use Label size with `--font-mono`.
- Timestamps in alerts use Label size with `--font-mono`.

## Color

Inherited from Phase 1 semantic tokens with dashboard-specific additions:

| Role | Semantic Token | Usage |
|------|---------------|-------|
| Dominant (60%) | `--background` / `--card` | Dashboard background, metric card surfaces |
| Secondary (30%) | `--muted` / `--accent` | Alert feed background, section dividers, inactive stall indicators |
| Accent (10%) | `--primary` | See "Accent reserved for" below |
| Destructive | `--destructive` | Critical alerts only |
| Success | `--success` (oklch(0.72 0.19 142)) | Active stall indicators, healthy system status, positive rate trends |
| Warning | `--warning` (oklch(0.82 0.17 85)) | Warning alerts (duplicate scans, offline devices) |

**Accent reserved for (Phase 9 additions):**
- SSE connection status indicator (connected = success, reconnecting = warning, disconnected = destructive)
- Active tab indicator on dashboard sections
- Hero attendance counter progress bar fill

**Alert severity colors:**

| Severity | Color Token | Icon | Usage |
|----------|------------|------|-------|
| critical | `--destructive` | `AlertTriangle` | Counter mismatch, system health failure |
| warning | `--warning` | `AlertCircle` | Duplicate scans, retroactive rejections |
| info | `--muted-foreground` | `Info` | Offline device detected, high scan rate |

**Stall status indicators:**

| Status | Color | Visual |
|--------|-------|--------|
| Active (scanned in last 60s) | `--success` | Filled circle (8px) |
| Idle (no scan in 60s-5min) | `--warning` | Filled circle (8px) |
| Inactive (no scan in 5min+) | `--muted-foreground` | Outline circle (8px) |

```

|  [ Overview | Config | Categories | Vendors | Live ]         |
```

**Tab visibility:** The "Live" tab is visible only when `event.status === "live"`. For non-live events, the tab does not render.

## Interaction Patterns

### Dashboard Load Flow
1. Admin navigates to event detail page and clicks "Live" tab
2. Frontend creates EventSource connection to `GET /api/v1/events/:id/live`
3. Connection status bar shows "Reconnecting..." during handshake
4. Server sends initial `snapshot` event with full state
5. Dashboard renders all metric cards, food consumption, stall monitor, and alerts
6. Status bar updates to "Connected via SSE" with green indicator
7. "Last update" timer starts counting seconds since last received event

### Real-Time Update Flow
1. Scan occurs at a vendor stall
2. Go microservice publishes scan event to Redis Pub/Sub
3. SSE handler receives event, pushes `counters` and/or `alert` event to connected admins
4. Frontend receives SSE event, patches TanStack Query cache via `queryClient.setQueryData`
5. React components re-render with updated values
6. Counter numbers animate briefly (0.3s CSS transition on `opacity` -- fade-in effect, not counting animation)
7. "Last update" timer resets to "just now"

### Reconnection Flow
1. SSE connection drops (network issue, server restart)
2. EventSource API auto-reconnects (browser-native behavior)
3. Status bar shows "Reconnecting..." with warning color
4. On reconnect, server sends fresh `snapshot` event
5. Dashboard re-renders with current state
6. Status bar returns to "Connected via SSE"
7. If disconnected for >30s, show "Connection lost -- retrying" with destructive color

### Alert Feed Interaction
1. New alerts appear at top of feed with a subtle slide-in animation (translateY from -8px, 0.2s)
2. Feed is scrollable with max height 300px
3. "Clear All" removes all alerts from the UI (client-side only, no server call)
4. Alerts have no click action -- they are informational
5. Alert feed is capped at 50 items (oldest removed when exceeded)

### Stall Monitor Interaction
1. Stall rows are sorted by scan rate (busiest first)
2. Status dot color changes based on last scan recency (active/idle/inactive thresholds)
3. "See breakdown" text button on Active Stations card scrolls to Stall Monitor section
4. Stall list is scrollable with max 10 visible rows, scroll for more

## Accessibility Contract

Inherited from Phase 1 with dashboard-specific additions:

- SSE connection status is conveyed via both color AND text label (never color alone)
- Alert feed items include `role="log"` and `aria-live="polite"` for screen reader announcements
- New alerts use `aria-live="polite"` (not `assertive` -- dashboard alerts are informational, not urgent)
- Counter values use `aria-label` to provide context (e.g., `aria-label="32,450 guests checked in out of 60,000 invited"`)
- Progress bars use `role="progressbar"` with `aria-valuenow`, `aria-valuemin`, `aria-valuemax`
- Stall status dots have `aria-label` describing the status ("Active", "Idle", "Inactive")
- Tab navigation works across all dashboard sections
- All interactive elements (Clear All, See breakdown) are keyboard accessible

## State Management Contract

| State Type | Tool | Scope |
|------------|------|-------|
| Dashboard metrics (counters, stalls, alerts) | TanStack Query (`queryClient.setQueryData`) | Per-event, updated via SSE events |
| SSE connection status | React `useState` in `useSSE` hook | Per-dashboard instance |
| Alert feed items | React `useState` in `LiveDashboard` | Per-dashboard instance, max 50 items |
| Last update timestamp | React `useState` + `useEffect` interval | Per-dashboard instance |
| Stall sort order | Derived from data (computed, not stored) | Recomputed on each update |

**No Zustand needed** for Phase 9. All state is either SSE-driven (TanStack Query cache) or local component state. No cross-component shared UI state that would benefit from a store.

## Design Principles

**Aesthetic direction:** Information-dense operational dashboard. Inspired by Grafana's metric panels and Linear's real-time activity feeds. Dense but not cluttered -- each metric card has breathing room (16px padding) but the overall layout prioritizes showing all critical metrics above the fold on desktop.

**Differentiation:** The hero attendance counter with a progress bar is the visual anchor. It gives an instant "how's the event going?" answer. The stall monitor with colored status dots provides at-a-glance operational awareness.

**What to avoid:**
- Charting libraries (no recharts, no chart.js) -- simple progress bars and counter numbers are sufficient for v1. Charts add bundle weight and visual complexity without proportional value.
- Counting animations on number updates -- a brief opacity fade is sufficient. Counting up from 0 on every SSE event would be distracting.
- Notification sounds for dashboard alerts -- this is a monitoring view, not an alarm system.
- Auto-scrolling the alert feed -- admin controls their scroll position, new items appear at top.
